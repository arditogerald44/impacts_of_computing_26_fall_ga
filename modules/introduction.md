## 🧠 1. Theoretical Context: Materiality and the War on Bloat

To understand the architecture of this course, we must confront the political and material realities of how software is deployed in higher education. We look at the history of computing through a dual framework: Tim Berners-Lee’s original decentralized web protocol and modern **Minimal Computing** design principles.

### 🛑 A. Digital Redlining vs. Minimal Computing
Modern corporate Educational Technology (EdTech) platforms are frequently agents of **digital redlining**. By building heavy, bloated web interfaces that require constant high-speed broadband connections, real-time tracking scripts, and up-to-date consumer hardware, institutions enforce invisible class boundaries that discriminate against specific student populations.

In direct response, we adopt a philosophy of **Minimal Computing**. Minimal computing asks fundamental questions:
1. What do students actually need?
2. What are the cultural and material realities of their remote learning spaces?
3. How can our educational infrastructure be made safer, more accessible, and more private?
4. How do we reduce our computational footprint to minimize electronic waste and environmental extraction?

```text
       [THE ARCHITECTURAL SPLIT]
       
  BLOATED INSTITUTIONAL PORTALS         OUR MINIMALIST SANDBOX
  ─────────────────────────────         ──────────────────────
  • High bandwidth dependencies         • Ultra-lightweight raw markdown text
  • Invisible telemetry analytics       • Zero background tracking scripts
  • Proprietary code enclosure          • Open-source Git version control
  • Perpetuates digital redlining       • Radical, democratized student access
```

### 🕸️ B. The 1989 Genesis Hook

This approach takes us straight back to the original architecture of the internet. When Tim Berners-Lee invented the World Wide Web on a **NeXTcube workstation** in 1989, his protocols (HTTP and HTML) were built on text-based openness. Every early browser was also configured as an editor. The web was meant to be a lightweight, decentralized conversation, not a hyper-centralized, corporate data trap.

By stripping our course environment down to raw Markdown files and terminal Git pushes, we aren't using an outdated system; we are running a conscious act of resistance. We are applying minimal computing principles to bypass corporate trackers, protect student privacy, and ensure that our course materials can run effortlessly on any machine on earth.

## 🛠️ Weekly Lab Evaluation Options

### 💻 Option A: The Developer Track (The Local Markdown Boilerplate)

1. Initialize your local Git workspace repository. Author a clean, optimized shell script that automatically parses plain-text data fields to generate a formatted HTML file entirely free of external JavaScript dependencies or styles.
2. **Deliverable:** Submit your script and an initialized repository state. Prove that your output file size remains under 5 Kilobytes (KB) while rendering flawlessly inside a terminal-based web browser environment.

### 🔍 Option B: The Analyst Track (The Digital Redlining and Network Audit)

1. Select a standard public-facing web platform or institutional portal enforced by an institution or business you interact with daily.
2. Run a comparative performance audit measuring it against the core principles of **Minimal Computing** and the definitions of **Digital Redlining**:
   - **The Bandwidth & Bloat Audit:** Use your browser's inspection panel to document the total page load size (in Megabytes), the number of network connections established, and the presence of third-party telemetry or tracking pixels.
   - **The Equity Access Diagnostic:** Analyze how the platform's heavy layout layout punishes a user with limited bandwidth or an older operating system.
3. **Deliverable:** Submit a forensic analysis to your repository. Compare the platform's data footprint against our course's markdown repository structure, defending whether bloated design represents an accidental software choice or an active tool of data exclusion.
